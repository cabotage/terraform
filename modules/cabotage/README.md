# cabotage

Terraform module that deploys the full [Cabotage](https://github.com/cabotage/cabotage-app) platform onto an existing Kubernetes cluster. This includes the application itself, its backing services (Consul, Vault, Postgres, Redis, object storage), a private container registry, an internal PKI, and resident monitoring.

Designed to be used alongside [`cabotage-eks`](../cabotage-eks), which provisions the underlying EKS cluster and VPC.

## What it creates

### Ingress & TLS

- **Traefik** — Ingress controller with proxy protocol support, forwarded header trust, Datadog + Prometheus metrics, and the Cabotage root CA loaded for backend TLS verification. Service type is configurable (NodePort, LoadBalancer, or AWS NLB).
- **cert-manager** — Certificate lifecycle management with CSI driver for pod-mounted certs. Includes a `letsencrypt` ClusterIssuer (real ACME in production, local CA in dev via `enable_pebble_letsencrypt`).

### Internal PKI

- **Root CA** — Generated locally via script; the private key never enters Kubernetes or Terraform state.
- **Intermediate CAs** — Two intermediate CAs (`certificate-approver` and `operators`) whose keys are generated by cert-manager, then signed offline with the root CA and patched back into the cluster.
- **CA admission webhook** — Mutating webhook that injects the Cabotage CA bundle into pods so they trust internal TLS certificates.

### Service discovery & secrets

- **Consul** — StatefulSet with gossip encryption, ACL bootstrap, and mTLS via cert-manager CSI volumes. Used as Vault's storage backend.
- **Vault** — StatefulSet with Consul client sidecar. Supports AWS KMS auto-unseal (production) or a dev sidecar auto-unseal. Bootstrap script initializes Vault, unseals all pods, enables auth backends, mounts secrets engines, and configures an internal PKI CA.

### Data stores

- **PostgreSQL** — CloudNativePG (CNPG) operator + cluster with TLS certificates signed by the operators CA.
- **Redis** — OT-Container-Kit Redis operator + cluster with TLS and password authentication.
- **RustFS** — S3-compatible object storage (replaces MinIO). StatefulSet with auto-provisioned buckets for the app, registry, Loki, and Mimir.

### Application

- **Cabotage app** — Web, worker, and worker-beat deployments. Bootstrap scripts configure Vault policies, Consul policies, and a transit backend. Post-deploy configuration patches in database/Redis/S3 connection details and runs migrations.
- **Enrollment operator** — Kopf-based operator that manages `CabotageEnrollment` custom resources for service identity provisioning (Vault/Consul credentials).
- **Container registry** — Distribution-based registry with RustFS backend storage, garbage collection CronJob, and signing certificate integration with the Cabotage app.
- **GitHub App integration** (optional) — Configures a GitHub App secret for webhook-driven builds when `github_app_id` is set. Supports OAuth login via `github_oauth_only` and `github_oauth_allowed_orgs`, with client credentials read from the secrets directory.

### Monitoring

- **Alloy** — DaemonSet log/metrics collector.
- **Loki** — Log aggregation (read/write/backend StatefulSets) with RustFS-backed S3 storage.
- **Mimir** — Metrics storage (read/write/backend StatefulSets) with RustFS-backed S3 storage and alerting rules.

## Usage

```hcl
module "cabotage" {
  source = "./modules/cabotage"

  cluster_identifier      = module.cabotage_eks.cluster_name
  kube_context            = "arn:aws:eks:us-east-2:123456789012:cluster/cabotage-prod"
  forwarded_headers_cidrs = ["10.0.0.0/16"]
  proxy_protocol_cidrs    = ["10.0.0.0/16"]

  cabotage_app_hostname = "cabotage.example.com"

  # Traefik behind AWS NLB
  traefik_aws_lb = true

  # Vault KMS auto-unseal (from cabotage-eks outputs)
  vault_auto_unseal_kms_key_id = module.cabotage_eks.vault_unseal_kms_key_id
  vault_auto_unseal_role_arn   = module.cabotage_eks.vault_unseal_irsa_role_arn
  vault_auto_unseal_region     = "us-east-2"

  # Optional: GitHub App for webhook builds
  github_app_id = "12345"

  # Optional: GitHub OAuth login
  github_oauth_only         = true
  github_oauth_allowed_orgs = "cabotage"
}
```

## Prerequisites

- A Kubernetes cluster with the `traefik` namespace already created (done by `cabotage-eks`)
- `kubectl` configured with the context specified in `kube_context`
- `openssl` available locally (for root CA generation and intermediate signing)
- A `secrets_dir` directory for storing bootstrap tokens (defaults to `.secrets`)

## Requirements

| Name | Version |
|------|---------|
| kubectl provider (`alekc/kubectl`) | >= 2.1 |
| local provider | >= 2.5 |

The module also uses the `kubernetes`, `helm`, `random`, and `null` providers (version constraints inherited from the root module).

## Bootstrap order

The module encodes a dependency chain that mirrors the manual cluster setup:

1. Traefik (ingress controller)
2. Namespace + cert-manager + CSI driver
3. Root CA generation + intermediate CA signing
4. CA admission webhook
5. Consul StatefulSet + ACL bootstrap
6. Vault StatefulSet + init/unseal/configure
7. CNPG operator + Redis operator
8. Enrollment operator + CRD
9. RustFS + bucket creation
10. Postgres cluster + Redis cluster
11. Cabotage app (web/worker/worker-beat) + enrollment
12. Container registry
13. Monitoring stack (Alloy, Loki, Mimir)

## Secrets

Bootstrap scripts store sensitive material in `secrets_dir` (default `.secrets/`):

| File | Contents |
|------|----------|
| `consul-management-token` | Consul ACL management token |
| `vault-bootstrap-token` | Vault root token |
| `vault-unseal-key` | Vault unseal key (manual unseal) |
| `vault-recovery-key` | Vault recovery key (KMS auto-unseal) |
| `github-app-private-key.pem` | GitHub App private key (user-provided) |
| `github-webhook-secret` | GitHub webhook secret (user-provided) |
| `github-app-client-id` | GitHub App OAuth client ID (user-provided, optional) |
| `github-app-client-secret` | GitHub App OAuth client secret (user-provided, optional) |

The root CA private key is also stored locally and never enters Kubernetes or Terraform state. For production, use Shamir key shares with PGP encryption for Vault unseal keys.
